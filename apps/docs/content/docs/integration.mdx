# Integration Guide

The registry serves different roles depending on who you are. This page covers integration patterns for each.

## As a Funder

You don't need to know the registry exists. Compute the deposit address for any identifier and transfer tokens directly to it — before or after the identifier is claimed.

```ts
import { CanonicalRegistrySDK } from "@ethereum-canonical-registry/sdk";

const sdk = new CanonicalRegistrySDK();
const { depositAddress } = await sdk.registry.resolve("github.com/org/repo");

await token.transfer(depositAddress, amount);
```

Funds accumulate at the deposit address until the identifier's owner claims and executes a transfer.

## As an On-Chain Consumer

The minimal interface for smart contracts that need to resolve identifiers:

```solidity
import {ICanonicalRegistry} from "@ethereum-canonical-registry/contracts/contracts/ICanonicalRegistry.sol";

bytes32 id = keccak256(abi.encode("github", "org/repo"));
address owner = ICanonicalRegistry(registry).ownerOf(id);

if (owner != address(0)) {
    // Identifier is claimed — send directly to owner
    token.transfer(owner, amount);
} else {
    // Identifier is unclaimed — send to deposit address for later retrieval
    token.transfer(depositAddress, amount);
}
```

`ownerOf` resolves through aliases transparently — no special handling needed for linked identifiers.

## As a Distribution Protocol

For protocols building allocation systems — grant programs, funding distributions, dependency tooling — the recommended flow at allocation creation time:

- **If claimed:** use the registrant address directly as the allocation recipient.
- **If unclaimed:** use the deposit address as the allocation recipient. Funds accumulate in the identity account until the identifier is claimed, at which point the owner can execute transfers.

The resolved address is stored as a plain Ethereum address in the upstream protocol. No knowledge of the registry is required at distribution time.

```ts
const state = await sdk.registry.resolve("github.com/org/repo", tokenAddress);

const recipient = state.owner ?? state.depositAddress;
```

## As a Registrant (Claiming and Executing)

1. **Generate a proof** via the web API for your namespace (GitHub OAuth or DNS TXT record).
2. **Claim** — submits the proof on-chain and deploys the identity account:

```ts
await sdk.registry.claim("github.com/org/repo", proof);
```

3. **Execute** — interact with your identity account using `execute`. The SDK provides helpers for common operations:

```ts
// Transfer ERC-20 tokens to your wallet
const { target, data } = sdk.account.encodeERC20Transfer(
  tokenAddress,
  myWallet,
  amount,
);
await sdk.account.execute(depositAddress, target, data);

// Withdraw from a Splits Warehouse
const call = sdk.account.encodeWarehouseWithdraw(
  warehouseAddress,
  depositAddress,
  tokenAddress,
);
await sdk.account.execute(depositAddress, call.target, call.data);
```

Only the registered owner can call `execute` — the identity account verifies ownership through the registry.

## Adding a New Namespace

Deploy a verifier implementing `IVerifier` and propose it to the registry admin for registration. Since the registry is a shared deployment, namespace registration is controlled by registry governance — anyone can build and propose a verifier, but `setVerifier` requires the admin key.

```solidity
interface IVerifier {
    function verify(bytes32 id, address claimant, bytes calldata proof) external returns (bool);
}

// Called by registry admin
registry.setVerifier("npm", address(npmVerifier));
```

Any `verify` implementation is valid — oracle-signed, ZK proof, on-chain attestation. Existing namespaces are unaffected.
