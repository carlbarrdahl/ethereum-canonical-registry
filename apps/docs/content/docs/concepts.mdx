# Concepts

The Canonical Registry is built on a small set of composable primitives. This page explains the mental model. For precise behavior, see [Requirements](/docs/requirements).

## Identifiers

An **identifier** is a bytes32 hash derived from a `(namespace, canonicalString)` pair:

```
id = keccak256(abi.encode(namespace, canonicalString))
```

Examples:
- `keccak256(abi.encode("github", "org/repo"))` — a GitHub repository
- `keccak256(abi.encode("dns", "example.com"))` — a DNS domain

`abi.encode` (not `abi.encodePacked`) is used to prevent collisions between pairs that share a common packed concatenation.

You can compute this locally — no RPC call needed:

```typescript
import { toId, canonicalise } from "@ethereum-canonical-registry/sdk";

const id = toId("github", canonicalise("Org/Repo")); // lowercased to "org/repo"
```

## Namespaces

A **namespace** scopes identifiers to a class of off-chain identity. Each namespace has its own canonicalisation rules and a registered verifier contract.

| Namespace | Canonical string | Example |
|---|---|---|
| `github` | `owner/repo` (lowercase) | `org/repo` |
| `dns` | apex domain (lowercase, no `www`) | `example.com` |

The contract does not enforce canonicalisation — it hashes whatever strings are passed. Consistency between funders and claimants is the caller's responsibility. The SDK's `canonicalise()` function applies the baseline rule (lowercase + trim).

## Deposit Addresses

Every identifier has a **deterministic deposit address** — a `ClaimableEscrow` proxy derived via CREATE2. The address is:

- **Stable**: computed from the identifier alone, independent of whether the escrow is deployed
- **Computable offline**: derivable from the identifier and registry address without an RPC call
- **Upgrade-safe**: address does not change when the escrow implementation is upgraded via the beacon

Funders send ERC-20 tokens directly to this address. Funds accumulate whether or not the identifier has been claimed.

The SDK handles address derivation internally:

```typescript
const state = await sdk.registry.resolveIdentifier("github", "org/repo", tokenAddress);
// state.depositAddress — the deterministic escrow address
```

## Claiming

To **claim** an identifier, the owner submits a cryptographic proof to the registry. The registry delegates verification to the namespace's registered verifier contract.

```
Owner → registry.claim(namespace, canonicalString, proof)
             │
             ▼
     verifier.verify(id, msg.sender, proof)
             │
       valid? │
             ▼
    owners[id] = msg.sender
    emit Claimed(...)
    deploy escrow if needed
```

A claim is rejected if:
- The identifier is already claimed
- No verifier is registered for the namespace
- The proof is invalid (wrong signer, expired, wrong claimant)

## Verifiers

A **verifier** is a contract that implements `IVerifier.verify(id, claimant, proof)`. The registry owner registers one verifier per namespace via `setVerifier`.

The current oracle verifier (`OracleVerifier`) uses EIP-712 typed signatures. The backend oracle:
1. Confirms ownership off-chain (GitHub OAuth, DNS TXT record)
2. Signs `{id, claimant, expiry}` with a trusted key
3. Returns the signature as the `proof` bytes

The EIP-712 domain binds the proof to a specific registry address and chain ID, preventing cross-deployment and cross-chain replay. Proofs expire to prevent indefinite replay.

```
DnsVerifier    GitHubVerifier
     │               │
     └───────┬────────┘
             │
        OracleVerifier (base)
             │
       verify(id, claimant, proof)
             │
    recover signer from EIP-712 sig
    check expiry + signer == trustedSigner
```

## Escrows

A **ClaimableEscrow** is a per-identifier `BeaconProxy` deployed via CREATE2. It:
- Holds ERC-20 tokens deposited by funders
- Integrates with the Splits Warehouse (pulls warehouse balance before transferring)
- Routes all withdrawals to the current registered owner in the registry

The escrow is deployed lazily — automatically when the owner claims, or by anyone calling `deployEscrow(id)`. Funders can send tokens to the deposit address before it's deployed; the address is always the same.

```typescript
// Anyone can trigger withdrawal — funds go to the owner, not the caller
await sdk.escrow.withdraw(escrowAddress, tokenAddress);
```

## Aliases

**Aliases** link multiple identifiers to a single owner, so `ownerOf(aliasId)` resolves to the primary owner. This allows a GitHub repo and its associated DNS domain to share a single identity.

```typescript
// Link "dns:example.com" as an alias for "github:org/repo"
await sdk.registry.linkIds(primaryId, [aliasId]);

// Both now resolve to the same owner
await sdk.registry.ownerOf(primaryId); // → 0x...
await sdk.registry.ownerOf(aliasId);   // → same 0x...
```

Rules:
- Only one level of indirection — no alias chains
- All identifiers must already be claimed by the same address
- Alias escrows remain separate; funds in each accrue independently
- Unlinking restores each identifier to independent ownership

## SplitsWarehouse

The **SplitsWarehouse** (from [0xSplits](https://splits.org/)) is an ERC-6909 multi-token vault integrated into `ClaimableEscrow`. When a funder deposits via `warehouse.deposit`, the escrowed balance shows up in the warehouse. `withdraw` pulls the warehouse balance out before sending the combined total to the owner.

This integration means funders can use Splits' batch deposit infrastructure to fund multiple escrows in one transaction.

## Resolving an Identifier

The SDK provides a convenience method to resolve full state in parallel:

```typescript
const state = await sdk.registry.resolveIdentifier("github", "org/repo", tokenAddress);
// state.id             — bytes32 identifier
// state.depositAddress — where funders send tokens
// state.owner          — null if unclaimed
// state.balance        — current ERC-20 balance at deposit address
```

## Design Principles

| Principle | Implementation |
|---|---|
| **No prior setup required** | Funders can send tokens before the owner has ever interacted with the protocol |
| **Permissionless** | Anyone can deploy escrows, send tokens, or call `withdraw` |
| **Non-custodial** | Only the registered owner can receive funds from `withdraw` |
| **Upgradeable** | Beacon pattern allows fixing bugs without changing any deposit address |
| **Pluggable verification** | Verifiers are swappable per namespace — DNSSEC on-chain verification can replace the oracle model later |
