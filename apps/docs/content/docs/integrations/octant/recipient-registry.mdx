# Recipient Registry

Both protocols need to verify recipients and display project metadata. A shared registry reduces duplication and improves discovery.

## Shared Data

| Data | Purpose |
|------|---------|
| Address | Recipient wallet/multisig |
| Name/description | Display in UIs |
| Verification | KYC, attestations, proof of work |
| Categories | Discovery and filtering |
| Funding history | Cross-protocol reputation |

## Technical Integration

**Option A: Shared on-chain registry**

```solidity
interface IRecipientRegistry {
    struct Recipient {
        address wallet;
        string metadataURI;  // IPFS with name, description, logo
        bytes32[] attestations;  // EAS attestation UIDs
    }
    
    function register(Recipient calldata recipient) external;
    function isVerified(address wallet) external view returns (bool);
}
```

Both Curator and Octant query the same registry for recipient data.

**Option B: Attestation-based (EAS)**

```solidity
// Schema: recipient verification
bytes32 constant RECIPIENT_SCHEMA = keccak256("address wallet, string name, bool verified");

// Either protocol can issue attestations
// Both protocols query EAS for attestations
function isVerified(address wallet) external view returns (bool) {
    return eas.hasAttestation(wallet, RECIPIENT_SCHEMA, trustedAttester);
}
```

**Option C: Indexer aggregation**

```typescript
// Aggregate from multiple sources
const recipient = {
  ...await curatorIndexer.recipient(address),
  ...await octantApi.project(address),
  ...await eas.attestations(address),
};
```

## Why This Works

- Single verification process for recipients
- Projects discoverable across both ecosystems
- Funding history aggregates cross-protocol reputation
- Reduces friction for new recipients

## Considerations

- Governance: Who can verify/unverify recipients?
- Standards: Agree on metadata schema
- Privacy: What data is public vs private?
- Sybil resistance: Shared approach to duplicate detection
