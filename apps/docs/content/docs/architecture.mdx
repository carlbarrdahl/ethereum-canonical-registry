# Architecture

## System Overview

```
┌──────────────────────────────────────────────────────────────────────┐
│                          ON-CHAIN                                    │
│                                                                      │
│  CanonicalRegistry                                                   │
│       │                                                              │
│       ├── verifiers[namespace] ──▶ DnsVerifier / GitHubVerifier     │
│       │                              (OracleVerifier base)           │
│       │                                                              │
│       └── beacon ──▶ ClaimableEscrow (impl)                         │
│                            │                                         │
│              CREATE2 proxies per identifier                          │
│                            │                                         │
│  Funders ──(ERC-20)──▶  0x4a3b... (escrow proxy)                   │
│                            │                                         │
│                       withdrawTo() ──▶ registered owner             │
│                                                                      │
│  SplitsWarehouse ◀── optional deposit path                          │
│                                                                      │
├──────────────────────────────────────────────────────────────────────┤
│                         OFF-CHAIN                                    │
│                                                                      │
│  Oracle backend ──signs proofs──▶ Claimants                         │
│                                                                      │
│  Contracts ──events──▶ Indexer (Ponder) ◀──queries──▶ SDK ──▶ Web  │
└──────────────────────────────────────────────────────────────────────┘
```

### On-Chain Layer

| Contract | Purpose | Pattern |
|---|---|---|
| **CanonicalRegistry** | Manages identifier ownership, escrow deployment, verifier registry | Ownable singleton |
| **ClaimableEscrow** | Holds funds per identifier; routes withdrawals to owner | BeaconProxy per identifier |
| **UpgradeableBeacon** | Points all escrow proxies to the current implementation | OZ beacon |
| **OracleVerifier** | Base EIP-712 verifier using a trusted backend signer | Abstract base |
| **DnsVerifier** | Verifies DNS TXT record ownership via oracle | Extends OracleVerifier |
| **GitHubVerifier** | Verifies GitHub repo ownership via oracle | Extends OracleVerifier |
| **SplitsWarehouse** | ERC-6909 multi-token vault for escrowed balances | External singleton (0xSplits) |

### Off-Chain Layer

| Component | Purpose |
|---|---|
| **Oracle backend** | Confirms off-chain ownership (DNS TXT, GitHub OAuth), signs EIP-712 proofs |
| **Indexer** (Ponder) | Processes contract events, exposes queryable state via GraphQL + REST |
| **SDK** (`@ethereum-canonical-registry/sdk`) | TypeScript client for contracts and indexer, React hooks, pure utils |
| **Web App** (`@ethereum-canonical-registry/web`) | Next.js frontend |

## Capital Flow

```
Funder ──(ERC-20 transfer)──▶ 0x4a3b... (ClaimableEscrow proxy)
                                        │
                         Owner claims identifier
                                        │
                         withdrawTo(token) called (by anyone)
                                        │
                                        ▼
                              Registered owner receives funds
```

1. Funder sends ERC-20 tokens to the deterministic deposit address
2. Tokens accumulate in the `ClaimableEscrow` proxy (or Splits Warehouse if deposited there)
3. After the owner claims the identifier with a valid proof, anyone can call `withdrawTo(token)`
4. Funds route exclusively to the current registered owner — the caller receives nothing

The deposit address is stable and computable before the escrow proxy is deployed. Funders do not need to wait for the owner to claim.

## Security Model

### Trust Assumptions

| Actor | Trust Level | Capabilities |
|---|---|---|
| **Registry owner** | Trusted admin | Can update verifiers, upgrade escrow impl via beacon |
| **Oracle signer** | Trusted for proofs | Signs claim proofs after off-chain verification; cannot affect existing ownership |
| **Claimants** | Must prove ownership | Can claim only with a valid, unexpired oracle proof |
| **Funders** | Trustless | Send tokens to deposit addresses; receive nothing back |
| **Withdrawers** | Permissionless | Call `withdrawTo`; funds always go to registered owner, not caller |

### Non-Custodial Properties

`ClaimableEscrow` has **no function to redirect funds to an arbitrary address**. The only exit path for funds is `withdrawTo(token)`, which sends to `registry.ownerOf(id)`. The registry owner can change verifiers or upgrade the escrow implementation, but cannot redirect funds from an existing escrow.

`withdrawTo` is **permissionless** — anyone can trigger it. This prevents funds from being locked by an inactive owner.

### Verifier Security

Oracle proofs are EIP-712 typed data bound to:
- `registry` address (prevents cross-deployment replay)
- `chainId` (prevents cross-chain replay)
- `id` (specific identifier)
- `claimant` (specific wallet)
- `expiry` (time-limited)

The trusted signer key can be rotated via `setTrustedSigner` on the verifier. Existing claims are not affected by a signer rotation.

### Escrow Upgradeability

The beacon pattern allows the registry owner to fix bugs in the escrow implementation without changing any deposit addresses. All proxies automatically point to the new implementation after an upgrade. The upgrade cannot change which identifier an escrow belongs to or redirect withdrawals to a different owner — those invariants are enforced by `registry.ownerOf(id)`.

### Known Limitations (POC)

| Limitation | Risk | Production Solution |
|---|---|---|
| Oracle-based verification | Requires trusting the oracle backend | Replace with DNSSEC on-chain verification via ENS DNSSECImpl |
| Single trusted signer | Key compromise allows fraudulent claims | Threshold signatures or HSM |
| No claim timelock | Fresh claims immediately enable withdrawal | Add claim cooldown for large deposits |
| Fee-on-transfer tokens | Balance calculations off | Explicit token allowlist or deposit accounting |

## Technical Decisions

### CREATE2 for Deposit Addresses

Using the identifier as the CREATE2 salt means the deposit address is:
- Derivable from the identifier alone (no on-chain lookup)
- Stable across re-deployments (same identifier → same address)
- Unique per identifier (no collisions)

Anyone can compute the address with `keccak256(abi.encode(namespace, canonicalString))` and the BeaconProxy initcode — the SDK's `toId()` utility handles this.

### BeaconProxy for Escrows

The beacon pattern is chosen over:
- **Plain clones (EIP-1167)**: non-upgradeable; a bug in escrow logic cannot be fixed without redeploying all proxies to new addresses
- **Individual transparent proxies**: each proxy stores its own admin/implementation slot, costing more gas and providing no groupwide upgrade path
- **Upgradeable proxies per clone**: same upgradeability as beacon but storage layout is shared across all instances, reducing per-proxy overhead

### abi.encode vs. abi.encodePacked

`abi.encode` prefixes each argument with its length, preventing `("githu", "borg/repo")` from hashing identically to `("github", "org/repo")`. `abi.encodePacked` would produce the same bytes for both.

### Namespace-Scoped Verifiers

A single `mapping(bytes32 => address) public verifiers` (keyed by `keccak256(bytes(namespace))`) allows:
- Independent verification logic per namespace
- Upgrades to one namespace without affecting others (e.g., migrate DNS to on-chain DNSSEC)
- Adding new namespaces (npm packages, Twitter handles, etc.) without changing the registry

### SplitsWarehouse Integration

Escrows integrate with Splits Warehouse so funders can use Splits' batch deposit tools. `withdrawTo` first pulls any warehouse balance, then transfers the combined ERC-20 balance to the owner — a single call handles both paths.
