### On-Chain Layer

| Contract              | Purpose                                                             | Pattern                    |
| --------------------- | ------------------------------------------------------------------- | -------------------------- |
| **CanonicalRegistry** | Manages identifier ownership, account deployment, verifier registry | Ownable singleton          |
| **IdentityAccount**   | Per-identifier smart account; owner can `execute` arbitrary calls   | BeaconProxy per identifier |
| **UpgradeableBeacon** | Points all account proxies to the current implementation            | OZ beacon                  |
| **OracleVerifier**    | Base EIP-712 verifier using a trusted backend signer                | Abstract base              |
| **DnsVerifier**       | Verifies DNS TXT record ownership via oracle                        | Extends OracleVerifier     |
| **GitHubVerifier**    | Verifies GitHub repo ownership via oracle                           | Extends OracleVerifier     |

### Off-Chain Layer

| Component                                        | Purpose                                                                    |
| ------------------------------------------------ | -------------------------------------------------------------------------- |
| **Oracle backend**                               | Confirms off-chain ownership (DNS TXT, GitHub OAuth), signs EIP-712 proofs |
| **Indexer** (Ponder)                             | Processes contract events, exposes queryable state via GraphQL + REST      |
| **SDK** (`@ethereum-canonical-registry/sdk`)     | TypeScript client for contracts and indexer, React hooks, pure utils       |
| **Web App** (`@ethereum-canonical-registry/web`) | Next.js frontend                                                           |

## Capital Flow

```
Funder в”Җв”Җ(ERC-20 transfer)в”Җв”Җв–¶ 0x4a3b... (IdentityAccount proxy)
                                        в”Ӯ
                         Owner claims identifier
                                        в”Ӯ
                         execute(target, data, value)
                                        в”Ӯ
                                        в–ј
                              Any on-chain interaction
                              (transfer tokens, call protocols, etc.)
```

1. Funder sends ERC-20 tokens to the deterministic deposit address
2. Tokens accumulate in the `IdentityAccount` proxy
3. After the owner claims the identifier with a valid proof, they can call `execute` to interact with any contract
4. The `execute` function checks `registry.ownerOf(id) == msg.sender` вҖ” only the registered owner can act

The deposit address is stable and computable before the account proxy is deployed. Funders do not need to wait for the owner to claim.

## Security Model

### Trust Assumptions

| Actor              | Trust Level          | Capabilities                                                                      |
| ------------------ | -------------------- | --------------------------------------------------------------------------------- |
| **Registry owner** | Trusted admin        | Can update verifiers, upgrade account impl via beacon                             |
| **Oracle signer**  | Trusted for proofs   | Signs claim proofs after off-chain verification; cannot affect existing ownership |
| **Claimants**      | Must prove ownership | Can claim only with a valid, unexpired oracle proof                               |
| **Funders**        | Trustless            | Send tokens to deposit addresses; receive nothing back                            |
| **Owner**          | Authenticated        | Only the registered owner can call `execute` on their identity account            |

### Non-Custodial Properties

`IdentityAccount` restricts `execute` to the registered owner вҖ” `registry.ownerOf(id) == msg.sender`. No other address can move funds or interact through the account. The registry owner can change verifiers or upgrade the account implementation, but the `ownerOf` check ensures only the legitimate owner can execute calls.

### Verifier Security

Oracle proofs are EIP-712 typed data bound to:

- `registry` address (prevents cross-deployment replay)
- `chainId` (prevents cross-chain replay)
- `id` (specific identifier)
- `claimant` (specific wallet)
- `expiry` (time-limited)

The trusted signer key can be rotated via `setTrustedSigner` on the verifier. Existing claims are not affected by a signer rotation.

### Account Upgradeability

The beacon pattern allows the registry owner to fix bugs in the account implementation via `upgradeAccountImplementation(newImpl)` without changing any deposit addresses. All proxies automatically point to the new implementation after an upgrade. The upgrade cannot change which identifier an account belongs to вҖ” that invariant is enforced by `registry.ownerOf(id)`.

### Known Limitations (POC)

| Limitation                | Risk                                      | Production Solution                                          |
| ------------------------- | ----------------------------------------- | ------------------------------------------------------------ |
| Oracle-based verification | Requires trusting the oracle backend      | Replace with DNSSEC on-chain verification via ENS DNSSECImpl |
| Single trusted signer     | Key compromise allows fraudulent claims   | Threshold signatures or HSM                                  |
| No claim timelock         | Fresh claims immediately enable execution | Add claim cooldown for large deposits                        |
| Fee-on-transfer tokens    | Balance calculations off                  | Explicit token allowlist or deposit accounting               |

## Technical Decisions

### CREATE2 for Deposit Addresses

Using the identifier as the CREATE2 salt means the deposit address is:

- Derivable from the identifier alone (no on-chain lookup)
- Stable across re-deployments (same identifier вҶ’ same address)
- Unique per identifier (no collisions)

Anyone can compute the address with `keccak256(abi.encode(namespace, canonicalString))` and the BeaconProxy initcode вҖ” the SDK handles this internally.

### BeaconProxy for Identity Accounts

The beacon pattern is chosen over:

- **Plain clones (EIP-1167)**: non-upgradeable; a bug in account logic cannot be fixed without redeploying all proxies to new addresses
- **Individual transparent proxies**: each proxy stores its own admin/implementation slot, costing more gas and providing no groupwide upgrade path
- **Upgradeable proxies per clone**: same upgradeability as beacon but storage layout is shared across all instances, reducing per-proxy overhead

### General-Purpose `execute`

Rather than building specific withdrawal or protocol integration functions into the account contract, the `execute(target, data, value)` pattern delegates all interaction logic to the caller. This means:

- No contract upgrade needed to support new protocols
- The account can interact with any contract deployed in the future
- The SDK provides helper functions (`encodeERC20Transfer`, `encodeWarehouseWithdraw`) for common operations

### abi.encode vs. abi.encodePacked

`abi.encode` prefixes each argument with its length, preventing `("githu", "borg/repo")` from hashing identically to `("github", "org/repo")`. `abi.encodePacked` would produce the same bytes for both.

### Namespace-Scoped Verifiers

A single `mapping(bytes32 => address) public verifiers` (keyed by `keccak256(bytes(namespace))`) allows:

- Independent verification logic per namespace
- Upgrades to one namespace without affecting others (e.g., migrate DNS to on-chain DNSSEC)
- Adding new namespaces (npm packages, Twitter handles, etc.) without changing the registry
