# Requirements

Structured requirements using [EARS notation](https://alistairmavin.com/ears/) (Easy Approach to Requirements Syntax). For conceptual background, see [Concepts](/docs/concepts). For system design, see [Architecture](/docs/architecture).

**Templates used:**

- **Ubiquitous**: The [system] shall [behavior].
- **Event-driven**: When [trigger], the [system] shall [behavior].
- **State-driven**: While [state], the [system] shall [behavior].
- **Unwanted behavior**: If [condition], then the [system] shall [behavior].

---

## 1. Identifier Derivation

**CR-ID-01** The registry shall derive an identifier `id` from a `(namespace, canonicalString)` pair as `keccak256(abi.encode(namespace, canonicalString))`.

**CR-ID-02** The registry shall use `abi.encode` (not `abi.encodePacked`) to prevent identifier collisions between namespace/string pairs that share a common packed concatenation.

---

## 2. Deterministic Deposit Addresses

**CR-ADDR-01** The registry shall assign every identifier a deterministic Ethereum address, derivable by anyone without making an on-chain call.

**CR-ADDR-02** The registry shall compute deposit addresses via CREATE2 using the identifier as the salt and the BeaconProxy initcode as the creation code.

**CR-ADDR-03** The deposit address shall be stable across identity account implementation upgrades. Only the delegated logic shall change; the proxy address shall remain fixed.

---

## 3. Claiming

**CR-CLAIM-01** When a claimant submits a valid proof for an unclaimed identifier, the registry shall record `owners[id] = msg.sender`.

**CR-CLAIM-02** If an identifier is already claimed, then the registry shall revert the claim transaction.

**CR-CLAIM-03** If no verifier is registered for the requested namespace, then the registry shall revert the claim transaction.

**CR-CLAIM-04** If the submitted proof is invalid (wrong signer, expired, wrong claimant), then the registry shall revert the claim transaction.

**CR-CLAIM-05** If the identifier is an active alias (its primary is still claimed), then the registry shall revert the claim transaction. If the alias is stale (primary was revoked), the registry shall clear the alias entry and allow the claim to proceed.

---

## 4. Identity Account Deployment

**CR-ACCT-01** When an identifier's identity account has not yet been deployed, any caller may deploy it by calling `deployAccount(id)`.

**CR-ACCT-02** If the identity account for an identifier has already been deployed, then `deployAccount` shall revert.

**CR-ACCT-03** The registry shall deploy each identity account as a `BeaconProxy` pointing to the `UpgradeableBeacon` owned by the registry.

---

## 5. Identity Account Execution

**CR-EXEC-01** When `execute(target, data, value)` is called on an `IdentityAccount`, the account shall forward the call to `target` with the given `data` and `value`.

**CR-EXEC-02** If `msg.sender` is not the registered owner of the account's identifier (`registry.ownerOf(id)`), then `execute` shall revert.

**CR-EXEC-03** If the forwarded call to `target` reverts, then `execute` shall revert.

**CR-EXEC-04** The identity account shall accept native ETH deposits via `receive()`.

---

## 6. Revocation

**CR-REV-01** When an identifier owner calls `revoke(namespace, canonicalString)`, the registry shall delete `owners[id]`, returning the identifier to unclaimed state.

**CR-REV-02** If `revoke` is called on an identifier that is an alias, then the registry shall revert.

**CR-REV-03** If `revoke` is called by an address that is not the current owner, then the registry shall revert.

---

## 7. Alias Linking

**CR-LINK-01** When an owner calls `linkIds(primaryId, aliasIds)` and owns all supplied identifiers, the registry shall set `aliases[aliasId] = primaryId` for each alias.

**CR-LINK-02** While an identifier has an alias link, `ownerOf(aliasId)` shall return the owner of the primary identifier.

**CR-LINK-03** If `primaryId` is itself an alias, then `linkIds` shall revert.

**CR-LINK-04** When an owner calls `unlinkIds(primaryId, aliasIds)`, the registry shall delete each alias entry and restore independent ownership.

---

## 8. Verification

**CR-VER-01** The registry shall delegate ownership verification to a verifier contract registered for the relevant namespace.

**CR-VER-02** The registry owner shall be able to register, replace, or remove a verifier for any namespace via `setVerifier`.

**CR-VER-03** An oracle verifier shall reject a proof where `block.timestamp >= expiry`.

**CR-VER-04** An oracle verifier shall reject a proof where the recovered signer does not match `trustedSigner`.

**CR-VER-05** An oracle verifier shall bind proofs to the `(registryAddress, chainId)` pair via an EIP-712 domain separator to prevent cross-deployment and cross-chain replay.

**CR-VER-06** The registry owner shall be able to rotate the trusted signer on any oracle verifier via `setTrustedSigner`.

---

## 9. Account Upgradeability

**CR-UPG-01** The registry shall own an `UpgradeableBeacon` to which all `IdentityAccount` proxies delegate.

**CR-UPG-02** Only the registry owner shall be able to upgrade the account implementation via `upgradeAccountImplementation(newImpl)`.

**CR-UPG-03** After an upgrade, all existing identity account proxy addresses shall remain unchanged.

---

## 10. Ownership Resolution

**CR-OWN-01** `ownerOf(id)` shall follow at most one level of alias indirection.

**CR-OWN-02** `ownerOf(id)` shall return `address(0)` for unclaimed identifiers.

---

## 11. Indexer

**REQ-I1** When `Claimed` is emitted, the Indexer shall upsert an `identifier` record with `namespace`, `canonicalString`, `owner`, and `claimedAt`.

**REQ-I2** When `Revoked` is emitted, the Indexer shall set `owner = null` and record `revokedAt` on the identifier.

**REQ-I3** When `Linked` is emitted, the Indexer shall insert an `identifier_alias` record linking `aliasId` to `primaryId`.

**REQ-I4** When `Unlinked` is emitted, the Indexer shall delete the corresponding `identifier_alias` record.

**REQ-I5** When `AccountDeployed` is emitted, the Indexer shall update the identifier record with `accountAddress`.

**REQ-I6** The Indexer shall expose `GET /api/stats`, `GET /api/identifiers/:namespace`, and `GET /api/owner/:address/identifiers`.

---

## 12. SDK

**REQ-SDK1** The `CanonicalRegistrySDK` shall auto-configure a `PublicClient` and `Indexer` from the wallet's chain ID, falling back to chain `31337` if no wallet is provided.

**REQ-SDK2** The SDK shall provide pure, RPC-free utilities: `toId(namespace, canonicalString)`, `canonicalise(value)`, `parseUrl(input)`, `formatIdentifier(namespace, canonicalString)`, and `parseIdentifier(identifier)`.

**REQ-SDK3** The SDK shall provide `registry.ownerOf(id)`, `registry.predictAddress(id)`, `registry.claim(namespace, canonicalString, proof)`, `registry.revoke(namespace, canonicalString)`, `registry.linkIds(primaryId, aliasIds)`, `registry.unlinkIds(primaryId, aliasIds)`, `registry.deployAccount(id)`, `registry.isAccountDeployed(id)`, `registry.resolve(input, token?)`, `registry.resolveIdentifier(namespace, canonicalString, token?)`, and `registry.setVerifier(namespace, verifier)`.

**REQ-SDK3a** `registry.resolve` shall accept any free-form identifier string (URLs, `namespace:value`, domain names) and return the same `IdentifierState` as `resolveIdentifier`.

**REQ-SDK4** The SDK shall provide `account.execute(accountAddress, target, data, value?)` for executing arbitrary calls through an identity account.

**REQ-SDK5** The SDK shall provide calldata encoding helpers: `account.encodeERC20Transfer(token, to, amount)`, `account.encodeWarehouseWithdraw(warehouse, owner, token)`, and `account.encodeERC721Transfer(nft, from, to, tokenId)`.

**REQ-SDK6** The SDK shall provide a `CanonicalRegistryProvider` React context and `useCanonicalRegistrySDK()` hook.

**REQ-SDK7** The SDK shall provide React query hooks: `useIdentifier`, `useIdentifiers`, `useAliases`, `useRegistryStats`.

**REQ-SDK8** The SDK shall provide React mutation hooks: `useClaim`, `useRevoke`, `useLinkIds`, `useUnlinkIds`, `useDeployAccount`, `useExecute`.

**REQ-SDK9** When a mutation hook succeeds, the SDK shall invalidate related query keys after a delay to allow indexer processing.

---

## 13. Data Model

**REQ-DM1** The Indexer shall maintain the following entities: `identifier`, `identifier_alias`.

**REQ-DM2** The `identifier` entity shall include: `id` (bytes32), `namespace`, `canonicalString`, `owner` (nullable), `accountAddress` (nullable), `claimedAt` (nullable), `revokedAt` (nullable), `createdAt`.

**REQ-DM3** The `identifier_alias` entity shall include: `id`, `aliasId`, `primaryId`, `linkedAt`.
